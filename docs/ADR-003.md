# ADR-003: github-catalog — UX/AX Improvements

**Status:** Accepted (P0 + P1 implemented; P2 deferred)
**Date:** 2026-06-18
**Environment:** WSL2 · Bash 5.0+ · jq 1.7 · git · gh (optional)
**Builds on:** [ADR-001](ADR-001-github-catalog-rewrite.md) (engine), [ADR-002](ADR-002.md) (UX/AX layer)

---

## Context

ADR-002 made the tool usable: one CLI, dual-auth, lifecycle tracking, a structured report. This ADR proposes the *next* increment of User Experience (UX) and Agent Experience (AX) polish. Every proposal is held to four constraints:

1. **No new core runtime dependency** — Bash 5 + jq + git only; `gh` stays in its bridge.
2. **Minimal surface** — prefer a flag on an existing verb over a new command; add a command only when the *question* is genuinely new.
3. **Append-only / read-only-GitHub invariants** from ADR-001/002 are untouched.
4. **The JSONL streams remain the contract** — every new human affordance must have a machine-readable equivalent.

Proposals are grouped by return-on-effort. **P0 and P1 are implemented** as of this revision. P2 remains deferred.

---

## P0 — highest ROI, smallest surface

### P0.1 `doctor` — preflight / self-diagnosis

**Problem.** The #1 friction in ADR-002 was dual-auth confusion (`gh` account vs. SSH key). Users discover it only *after* a failed sync.

**Proposal.** `./github-catalog doctor [owner] [--git-host HOST]` runs read-only checks and prints a checklist:

```
github-catalog doctor
  ✓ bash 5.2 (>= 5.0)
  ✓ jq 1.7.1 (>= 1.7)
  ✓ git 2.43
  ✓ gh 2.45 — authenticated as: alex (github.com)
  ⚠ ssh github-personal — key loaded, but `git ls-remote git@github-personal:…` not tested
  ✓ data/qobeat/ — inventory present (42 repos, last refresh 2026-06-17)
```

**Why.** Converts a runtime failure into a guided preflight. Directly targets the dual-auth gap. **Effort:** low (all checks are existing one-liners). **Risk:** none (read-only).

### P0.2 Final stdout summary line

**Problem.** The progress bar writes to stderr and the run summary lands in the log; on success, stdout is nearly silent and never tells the user the next step.

**Proposal.** After a sync, print one line to **stdout**:

```
Synced 42 repos for qobeat: 7 collected, 34 skipped (unchanged), 1 failed, 18 new commits.
Next: ./github-catalog report qobeat
```

**Why.** Closes the loop; gives agents a parseable one-liner. **Effort:** trivial (data already in the `run` record). **Risk:** none.

### P0.3 `report --format json`

**Problem.** The report is Markdown-only; agents must re-parse prose or read raw JSONL and re-implement the report's aggregation (latest-per-repo, dedup, status rollups).

**Proposal.** `./github-catalog report <owner> --format json|md` (default `md`). The JSON form emits the *same* aggregation the Markdown uses (it is already computed in jq) as a single object: overview counts, per-repo summary array, commit activity, semantics, errors.

**Why.** The report's value is its aggregation, not its formatting. Exposing JSON is near-free and high-AX. **Effort:** low (split the jq pipeline's data step from its render step). **Risk:** low.

### P0.4 Per-owner config file

**Problem.** Users repeat `--git-host github-personal --private` on every command for the same owner.

**Proposal.** Optional `data/<owner>/catalog.config` (flat `key=value`, parsed in ~5 lines of bash — **no TOML library**):

```
git_host=github-personal
visibility=private
parallel=8
```

Precedence: explicit flag > env var (`GITHUB_CATALOG_*`) > config file > built-in default. `clean` preserves it unless `--purge`.

**Why.** Removes per-invocation repetition without a new flag surface. **Effort:** low. **Risk:** low (precedence must be tested).

## P1 — useful, slightly larger

### P1.1 `status` — fast catalog overview (no report write)

`./github-catalog status [owner] [--format json]` prints a compact, read-only digest — owners cataloged, repo counts by status (active/deleted), last sync run, repos with errors — **without** generating or writing a report file. Distinct *question* from `report` ("what's the state of my catalog?" vs. "give me the document"), so it earns a command.

### P1.2 `sync --dry-run`

Resolve the repo list, run the `ls-remote` sentry, and print what *would* be collected / skipped / is unreachable — **no clone, no writes**. Essential before a large-owner sync. Reuses the sentry path; collection is the only step suppressed.

### P1.3 `--quiet` / `--verbose`

`--quiet` suppresses the progress bar (CI/non-TTY); `--verbose` mirrors structured log lines to stderr in real time. Auto-detect non-TTY → quiet by default. Low effort, removes the "is it hung?" ambiguity in pipelines.

### P1.4 Documented exit-code contract

Standardize and document exit codes so scripts/agents can branch deterministically:

| Code | Meaning |
|------|---------|
| 0 | success |
| 1 | partial failure (≥1 worker failed; catalog still written) |
| 2 | usage / bad arguments |
| 3 | precondition failed (missing `gh`, unreachable owner, no inventory) |

Today `1` is overloaded. Splitting it is a small change with real AX value.

## P2 — nice to have / longer horizon

### P2.1 Capability descriptor (`--help --json`)

Emit a machine-readable descriptor of commands, flags, record types, and `schema_version`. Complements the prose `MANIFEST.md` with structure an agent can self-discover, and becomes the natural backing for the MCP server in [ADR-005](ADR-005.md).

### P2.2 Shell completion

Bash/Zsh completion for commands, flags, and owners (sourced from `data/*/`). Pure-bash completion script; no dependency.

### P2.3 Inventory pagination beyond 1000

ADR-002 lists the `gh repo list --limit 1000` ceiling as a known negative. Add cursor pagination in the bridge for very large owners. Larger effort; defer until a real >1000-repo owner appears.

## Decision summary

| ID | Item | New surface | Effort | Priority |
|----|------|-------------|--------|----------|
| P0.1 | `doctor` preflight | new command | low | **P0** |
| P0.2 | stdout summary line | none | trivial | **P0** |
| P0.3 | `report --format json` | flag | low | **P0** |
| P0.4 | per-owner config file | file (no flag) | low | **P0** |
| P1.1 | `status` digest | new command | low | P1 |
| P1.2 | `sync --dry-run` | flag | low | P1 |
| P1.3 | `--quiet` / `--verbose` | flags | trivial | P1 |
| P1.4 | exit-code contract | none (semantics) | low | P1 |
| P2.1 | capability descriptor | flag | medium | P2 |
| P2.2 | shell completion | aux file | low | P2 |
| P2.3 | inventory pagination | internal | medium | P2 |

**Recommended first cut:** P0.1–P0.4. They are individually small, share the existing data, add at most one command (`doctor`), and each removes a concrete, observed pain point.

**Implemented (P0 + P1):** `doctor`, stdout summary, `report --format json`, `catalog.config`, `status`, `sync --dry-run`, `--quiet`/`--verbose`, exit-code contract.

**Deferred (P2):** capability descriptor (`--help --json`), shell completion, inventory pagination beyond 1000 repos.

---

## Consequences

**Positive.** Self-diagnosing onboarding; agents get JSON parity and a clean exit-code contract; less flag repetition. All within the zero-dependency boundary.

**Negative / watch.** Each new flag and command is surface area to test and document; the minimal-interface principle must gate additions (a `status` command is justified by a distinct question — a `--git-host` shorthand is not). Config-file precedence is the one subtle correctness risk and must be unit-tested.

**Out of scope (deferred to their own ADRs).** Repository classification and richer extraction → [ADR-004](ADR-004.md). Conversational/programmatic access via MCP → [ADR-005](ADR-005.md); the capability descriptor (P2.1) is the bridge to it.
