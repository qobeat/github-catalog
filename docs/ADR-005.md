# ADR-005: github-catalog-mcp — Model Context Protocol Server

**Status:** Proposed
**Date:** 2026-06-18
**Environment:** Catalog engine = WSL2 · Bash 5 · jq · git. MCP server = separate repo (`github-catalog-mcp`), own runtime.
**Builds on:** [ADR-001](ADR-001-github-catalog-rewrite.md), [ADR-002](ADR-002.md); surfaces [ADR-004](ADR-004.md) classification when present

---

## Context & goal

The catalog is a high-signal, append-only, *local* knowledge base about an owner's repositories — identity, semantics, history, and (per ADR-004) archetype classification. The goal of this ADR is to expose it over the **Model Context Protocol (MCP)** so agents can query and drive it conversationally and programmatically — *"which of my repos are MCP servers?"*, *"summarize qobeat's catalog"*, *"sync ados-\* then show what changed"*.

This will live in a **separate repository, `github-catalog-mcp`** (as the user intends), pinning a compatible catalog `schema_version`.

### Non-goals

- **Not** a reimplementation of the engine. The bash CLI + JSONL remain the single source of truth; the MCP server is a thin **adapter/transport**.
- **Not** a clone of [`github/github-mcp-server`](https://github.com/github/github-mcp-server). That server talks to the **live GitHub API** (issues, PRs, Actions). Ours serves the **local, historical, classified catalog** — offline-queryable, provenance-bearing, with archetype facets GitHub doesn't expose. They are **complementary**: github-mcp-server answers "what's on GitHub right now?"; github-catalog-mcp answers "what does my curated catalog know across time?".

---

## Decision

Build `github-catalog-mcp` as a **thin MCP server that wraps the existing CLI for effectful actions and reads the JSONL/schema directly for queries.** The engine's invariants (append-only, read-only-GitHub) are inherited for free — even a `sync` tool cannot mutate GitHub.

```
MCP client (Claude, IDE, agent)
        │  JSON-RPC over stdio
        ▼
github-catalog-mcp   ── reads ──▶  data/<owner>/*.jsonl   (queries: fast, in-proc/jq)
 (separate repo)     ── execs ──▶  ./github-catalog …      (effects: sync/refresh)
        │
        └─ pins catalog schema_version range; validates records; exposes ADR-004 classification
```

### Runtime choice

The zero-dependency constraint is for the **engine**, not its clients. The MCP server may use a real SDK.

| Option | Assessment |
|--------|------------|
| **TypeScript + `@modelcontextprotocol/sdk`** | **Recommended.** First-class SDK, stdio + Streamable HTTP, trivial distribution via `npx github-catalog-mcp`, easy JSON/JSONL handling. Shells out to `./github-catalog` for effects. |
| Python + MCP SDK | Viable alternative; good if the team prefers Python. Same shape (exec CLI for effects, read JSONL for queries). |
| Pure Bash MCP | **Rejected** — no mature MCP SDK; hand-rolling JSON-RPC framing/lifecycle is brittle and offers no benefit. |

Recommendation: **TypeScript**. It keeps the engine pristine and the server distributable.

### Read vs. effectful boundary

Tools are split and annotated so clients can reason about safety:

- **Read tools** — query JSONL only; `readOnlyHint: true`, cacheable, always available.
- **Effectful tools** (`sync`, `refresh`) — touch the network, write JSONL, may spawn parallel git; `readOnlyHint: false`, `openWorldHint: true`. **Off by default**, enabled via env (`GITHUB_CATALOG_MCP_ALLOW_WRITES=1`), and they inherit the *same* ambient auth (`gh` token, SSH keys) the CLI uses.
- `clean` is **not exposed** by default (destructive); if ever added, it is gated separately and never supports `all`.

## Tool catalog (minimal, typed, high-AX)

Names are verb-first and unambiguous; inputs are typed; outputs are the catalog's own JSON shapes (no bespoke reformatting).

### Read (always on)

| Tool | Input | Returns |
|------|-------|---------|
| `list_owners` | — | Cataloged owners + repo counts + last sync |
| `list_repos` | `owner`, `status?` | Repo summaries (latest snapshot per slug) |
| `get_repo` | `owner`, `slug` | Full latest `repo_snapshot` (incl. `classification`) |
| `search_repos` | `owner`, `archetype?`, `language?`, `key_file?`, `ai_related?`, `status?` | Filtered repo summaries — the archetype query surface from ADR-004 |
| `get_commits` | `owner`, `slug`, `limit?` | Recent `commit` records |
| `get_report` | `owner`, `format=md\|json` | The rendered report (ADR-003 P0.3 JSON form) |
| `get_run_history` | `owner` | `run` records (sync provenance) |

### Effectful (opt-in)

| Tool | Input | Maps to |
|------|-------|---------|
| `sync` | `owner`, `glob?`, `visibility?`, `git_host?`, `parallel?` | `./github-catalog sync …` |
| `refresh` | `target`, `visibility?` | `./github-catalog refresh …` |

### Resources (read-only context)

- `catalog://schema` → `docs/github-catalog.schema.json` (so the agent learns the record contract).
- `catalog://{owner}/snapshot` → latest snapshot per repo (JSON).
- `catalog://{owner}/report` → `reports/{owner}/latest.md`.

### Prompts (optional)

- `summarize_owner(owner)` — templated prompt that pulls the report + classification rollup.
- `find_repos(description)` — turns a natural-language ask into `search_repos` facets.

The effectful↔CLI mapping means the server never duplicates engine logic; queries hit JSONL directly for speed.

## Cross-cutting decisions

**Schema contract.** The server declares a supported catalog `schema_version` range (e.g. `>=1.2.0 <2.0.0`), validates records on read, and degrades gracefully when optional fields (e.g. ADR-004 `classification`) are absent — reporting `unclassified` rather than failing.

**Transport.** Start **stdio** (local, the common Claude/IDE case). Add **Streamable HTTP** later only if a shared/hosted catalog is needed.

**Auth & isolation.** Read tools need only filesystem access to the catalog root (passed as `--catalog-root` / env). Effectful tools inherit the CLI's `gh`/SSH auth and are off by default. The server is confined to the catalog root — no arbitrary path access.

**Optional LLM enrichment.** The *only* sanctioned home for the fuzzy, non-deterministic labels ADR-004 excluded from the core (free-text summaries, subjective maturity) is an explicitly-marked, opt-in enrichment tool here — never written back into the deterministic `classification` object, and clearly flagged as model-generated.

**Versioning & distribution.** Published from `github-catalog-mcp`; run via `npx github-catalog-mcp --catalog-root /path/to/github-catalog`. Client config:

```jsonc
{ "mcpServers": {
    "github-catalog": {
      "command": "npx",
      "args": ["-y", "github-catalog-mcp", "--catalog-root", "/home/alex/dev/github-catalog"],
      "env": { "GITHUB_CATALOG_MCP_ALLOW_WRITES": "0" }
} } }
```

## Options considered

| Decision point | Chosen | Rejected |
|----------------|--------|----------|
| Engine relationship | Thin wrapper (exec CLI + read JSONL) | Reimplement catalog logic in the server (drift, double-maintenance) |
| Runtime | TypeScript SDK | Pure-bash MCP (no SDK); embedding engine in server |
| Default safety | Read-only; writes opt-in | Writes on by default (footgun: network + parallel git from a chat turn) |
| Output shapes | Catalog's own JSON/JSONL | Bespoke MCP DTOs (needless mapping, schema drift) |
| Scope vs. github-mcp-server | Local historical catalog | Duplicating live-API tooling |

## Phased plan

1. **Scaffold** `github-catalog-mcp` (TS SDK, stdio), `--catalog-root`, schema-version check.
2. **Read tools** `list_owners`/`list_repos`/`get_repo`/`get_commits` over JSONL. Validate with the MCP Inspector.
3. **Resources** `catalog://schema` + `catalog://{owner}/report`.
4. **`search_repos`** (the ADR-004 archetype/facet query surface) + `get_report --format json`.
5. **Effectful tools** `sync`/`refresh` behind `GITHUB_CATALOG_MCP_ALLOW_WRITES`, with `readOnlyHint`/`openWorldHint` annotations.
6. **Prompts** + optional LLM enrichment tool (clearly marked, opt-in).

**Acceptance:** each tool round-trips through the MCP Inspector; read tools never write; effectful tools refuse to run unless explicitly enabled; absent `classification` degrades to `unclassified`.

---

## Consequences

**Positive.** Agents query and drive the catalog with a small, typed, safe tool surface; the engine stays the single source of truth (no logic duplication); read-only-by-default plus the engine's read-only-GitHub invariant make it safe to expose; classification (ADR-004) becomes a first-class query dimension; complementary to — not competing with — the official GitHub MCP server.

**Negative / watch.** A second repo and runtime to maintain and version against the catalog schema (mitigated by the pinned range + graceful degradation). Effectful tools spawn network/parallel-git work from a single call — must stay opt-in and annotated. Distribution/auth (SSH keys, `gh` token) for effectful tools needs clear docs so users don't expect writes to work in a locked-down sandbox.

**Dependencies.** Strongest once [ADR-003 P0.3](ADR-003.md) (`report --format json`) and [ADR-004](ADR-004.md) (`classification`) land — `get_report` and `search_repos` are thin over them. The server can ship read tools before those, then light up the richer surface as the engine gains it.
